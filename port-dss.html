Overview
Project Overview

Container terminals must coordinate berth availability, crane assignment and truck gate sequencing simultaneously, with each decision cascading into the next. Small and medium EU terminals typically manage this with spreadsheets or fixed rules because commercial Terminal Operating Systems cost millions of euros to license.

Port DSS closes that gap with three tightly integrated CP-SAT optimisation engines built on Google OR-Tools — a berth allocator, a truck gate scheduler, and a crane assignment optimiser — paired with a real-time React dashboard over WebSocket, a FastAPI/PostgreSQL backend and a Redis pub-sub event bus. The entire seven-service stack deploys with a single docker compose up command.

A live demonstration on seeded data covering eight vessels, six berths, six cranes and ten trucks shows the berth solver reaching an optimal solution in under 0.1 seconds with zero vessel conflicts and zero crane idle collisions, while an explainability layer surfaces solver reasoning in plain language to build operator trust.

Architecture
System Pipeline
Phase 01
Berth Allocation Solver
CP-SAT solver with LOA, draft and cargo-type compatibility constraints, no-overlap interval variables and a multi-objective cost balancing waiting time, idle time and throughput.
Phase 02
Truck Gate Scheduler
Assigns gate entry slots respecting per-slot capacity limits and aligning truck arrival with berth availability windows across a 24-hour, 15-minute-slot horizon.
Phase 03
Crane Assignment Optimiser
Maximises container moves across active berth allocations while enforcing no-overlap and physical reach constraints per crane.
Phase 04
Explainability Layer
Translates every solver decision into plain-language reasoning, closing the trust gap that has historically blocked optimisation tool adoption in port environments.
Phase 05
Real-Time Event Bus
Redis pub-sub channels broadcast domain events; a WebSocket relay pushes updates to every connected operator dashboard within one second.
Phase 06
Single-Command Deployment
A seven-service Docker Compose stack (frontend, backend, database, Redis, Prometheus) with health-checked startup ordering, live in under 5 seconds.

Key Contributions
Novel Contributions

• Three-phase CP-SAT pipeline — berth, truck and crane solvers run sequentially within a single atomic database transaction, so partial schedules are never committed if a downstream solver fails.

• Explainability-first design — every optimisation run produces a human-readable JSON explanation, directly addressing the trust barrier that has historically blocked optimisation adoption at smaller ports.

• Database-level scheduling integrity — a unique constraint on berth and start-time acts as a failsafe against race conditions, rejecting conflicting inserts even under concurrent API requests.

• Operator-in-the-loop control — proposed allocations remain editable and reversible; the system supports decisions rather than replacing them, with a full audit trail of overrides.
